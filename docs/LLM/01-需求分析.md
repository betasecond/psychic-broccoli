# CourseArk LLM 能力补充需求分析

## 1. 背景

当前 CourseArk 项目已经具备课程管理、考试评分、作业批改、学习进度追踪、课程知识库 RAG 问答等能力，但与论文选题“基于 RAG 与 LLM 的实训教学评辅闭环系统设计与实现”相比，存在两个关键偏差：

1. 课程大纲导入目前主要依赖正则规则解析，不能充分体现 LLM 能力。
2. 试题解析/生成接口名为 `ParseQuestionsWithAI`，但现有实现主要是本地规则解析，论文中不宜写成真正的 LLM 辅助生成。

因此，本次需求是在不大规模重构现有系统的前提下，补充两个轻量、可验证、与选题强相关的 LLM 能力：

1. LLM 课程大纲解析。
2. LLM 试题结构化生成/解析。

同时确认现有 RAG 问答能力的边界，并将论文口径调整为“初步评-辅闭环”，避免虚写“自动根据薄弱点生成专项练习”等当前项目尚未完整实现的能力。

需要特别说明：RAG 是系统已有的辅导能力，本轮开发重点不是重写 RAG，而是补齐 LLM 在“教学资源结构化处理”和“教师辅助出题”两个场景中的真实调用能力。

## 2. 目标

本次改造目标如下：

1. 让课程大纲解析接口具备真实 LLM 调用能力。
2. 让试题解析接口具备真实 LLM 调用能力。
3. LLM 调用失败、超时、无 Key、返回非 JSON、字段校验失败时，系统必须回退到现有规则解析逻辑。
4. 保持前端已有功能入口基本不变，尽量减少调用方式变化。
5. 后端必须对 LLM 输出进行字段级校验，避免脏数据直接进入后续导入流程。
6. 论文可真实描述为：“系统采用 LLM 解析 + 规则回退的稳健导入方案；系统实现了 LLM 辅助试题结构化生成，并由教师确认后入库；系统已具备/保留 RAG+LLM 的课程知识库可溯源问答。”

## 3. 非目标

以下内容不纳入本轮开发，避免临时扩张导致 bug：

1. 不实现“自动根据薄弱知识点生成专项练习”的完整闭环。
2. 不新增题目知识点标签体系。
3. 不新增错题归因、学生个性化推荐、练习入库和学生作答入口。
4. 不替换现有 RAG 知识库架构。
5. 不引入新的数据库表，除非后续设计文档明确证明必要。
6. 不把 LLM 生成结果直接写入考试题库，必须保留教师确认步骤。

## 4. 用户角色与使用场景

### 4.1 教师

教师是本次 LLM 能力的主要使用者。

核心场景：

1. 教师上传课程大纲文件，系统调用 LLM 解析为章节和课时结构。
2. 教师检查解析结果后，选择是否批量导入课程章节。
3. 教师上传题目文本或自然语言要求，系统调用 LLM 输出结构化题目。
4. 教师检查、修改、确认题目后，再写入考试题库。
5. 教师仍可使用现有规则解析作为兜底能力。

### 4.2 学生

学生不直接使用本轮新增的 LLM 大纲解析和试题生成能力。

学生相关能力仍以现有 RAG 问答为主：

1. 学生围绕课程资料提问。
2. 系统检索课程知识库片段。
3. LLM 基于召回片段生成回答。
4. 回答展示引用来源，便于学生追溯资料依据。

### 4.3 管理员

管理员可使用教师同等的 LLM 大纲解析和试题解析能力，用于系统维护、演示和数据初始化。

## 5. 功能需求

### FR-01 课程大纲 LLM 解析

接口：`POST /api/v1/courses/:id/parse-outline`

当前输入方式保持不变：

1. `multipart/form-data`
2. 字段名：`file`
3. 支持格式：`.txt`、`.md`
4. 文件大小上限：沿用现有 2MB 限制

新增行为：

1. 后端读取文件文本后，优先调用 LLM。
2. LLM 应返回严格 JSON。
3. JSON 字段校验通过后，返回解析出的章节和课时。
4. LLM 调用失败或校验失败时，自动回退到现有正则解析。
5. 响应中应增加解析模式字段，便于调试与测试。
6. 解析接口只负责“识别结构”，不直接写入章节/课时表。
7. 教师确认后，前端继续复用现有章节创建接口和课时创建接口完成批量导入，不新增大纲确认导入接口。

建议响应结构：

```json
{
  "chapters": [
    {
      "title": "第一章 Go 语言基础",
      "orderIndex": 1,
      "sections": [
        {
          "title": "变量与类型",
          "orderIndex": 1,
          "type": "VIDEO"
        }
      ]
    }
  ],
  "chapterCount": 1,
  "sectionCount": 1,
  "parseMode": "llm"
}
```

`parseMode` 可选值：

1. `llm`：LLM 解析成功。
2. `rule_fallback`：未配置 Key、LLM 调用失败、LLM 返回非法 JSON 或字段校验失败后，回退规则解析。

本轮默认策略为：先尝试 LLM，失败才规则解析。
后续如需提供“仅规则解析”的手动模式，可扩展 `rule_only`，但本轮不纳入验收范围。

### FR-02 大纲解析字段校验

LLM 输出必须满足：

1. 根对象包含 `chapters` 数组。
2. `chapters` 至少包含 1 个有效章节。
3. 每个章节 `title` 非空。
4. 每个章节 `orderIndex` 应为正整数；如果 LLM 返回异常，后端可重新编号。
5. `sections` 可为空数组。
6. 每个课时 `title` 非空。
7. 每个课时 `type` 只能是 `VIDEO` 或 `TEXT`；非法值按规则修正，无法修正时默认 `VIDEO`。
8. 每个课时 `orderIndex` 应为正整数；如果 LLM 返回异常，后端可重新编号。

### FR-03 试题 LLM 结构化生成/解析

接口：`POST /api/v1/exams/:id/parse-questions`

当前输入方式保持不变：

1. `multipart/form-data`
2. 字段名：`file`
3. 支持格式：`.txt`、`.md`、`.csv`
4. 文件大小上限：沿用现有 2MB 限制

新增行为：

1. 后端读取文本后，优先调用 LLM。
2. LLM 可处理两类输入：
   - 已有题目文本：抽取为结构化题目。
   - 自然语言要求：生成符合要求的结构化题目。
3. 本轮为了减少前端改动，自然语言要求也通过 `.txt` 或 `.md` 文件上传。例如文件内容可以是：“请生成 3 道 Go 并发单选题，每题 3 分，并给出正确答案。”后续如时间允许，可再新增前端文本输入框。
4. LLM 返回 JSON 后，后端进行字段校验。
5. 校验通过后返回题目列表，但不直接入库。
6. 教师在前端确认后，继续调用现有确认导入接口写入题库。
7. LLM 调用失败或 JSON 不合法时，回退到现有规则解析。

建议响应结构：

```json
{
  "questions": [
    {
      "type": "SINGLE_CHOICE",
      "stem": "Go 语言中用于启动协程的关键字是？",
      "options": ["go", "func", "defer", "chan"],
      "answer": "A",
      "score": 3,
      "confidence": 0.92,
      "issues": []
    }
  ],
  "count": 1,
  "parseMode": "llm"
}
```

### FR-04 试题字段校验

每道题必须满足：

1. `type` 必须是以下之一：
   - `SINGLE_CHOICE`
   - `MULTIPLE_CHOICE`
   - `TRUE_FALSE`
   - `SHORT_ANSWER`
2. `stem` 非空。
3. `score` 必须大于 0；缺失时使用默认分值。
4. 单选题必须至少 2 个选项，答案必须是一个有效选项字母。
5. 多选题必须至少 2 个选项，答案必须能归一化为多个选项字母，例如 `A,C`。
6. 判断题答案必须归一化为 `true` 或 `false`。
7. 简答题允许没有选项，但必须有参考答案。
8. 严重不合法题目必须过滤，不返回给前端。
9. 轻微问题题目可以保留，但必须写入 `issues`，由教师在确认导入前修正或删除。

严重不合法示例：

1. 题干为空。
2. 题型不在允许范围内且无法修正。
3. 选择题没有选项。
4. 判断题答案无法归一化为 `true` 或 `false`。

轻微问题示例：

1. 分值缺失，后端已使用默认分值补齐。
2. 多选题答案格式为 `AC`，后端已归一化为 `A,C`。
3. 选项数量偏少但仍满足最低要求。

### FR-05 LLM 失败回退规则

以下情况必须回退到规则解析：

1. 未配置 API Key，且请求头没有传入临时模型 Key。
2. LLM HTTP 请求失败。
3. LLM 响应状态码为错误。
4. LLM 返回空内容。
5. LLM 返回内容无法提取 JSON。
6. JSON 反序列化失败。
7. 字段校验后没有任何有效章节或题目。

回退后：

1. 如果规则解析成功，接口返回 `parseMode: "rule_fallback"`。
2. 如果规则解析也失败，接口返回 400，并提示检查文件格式或输入内容。

说明：`rule_fallback` 同时覆盖“未配置 Key”和“LLM 调用出错”两类情况。后续设计文档可增加 `fallbackReason` 作为调试字段，但本轮需求不强制前端展示。

### FR-06 临时模型 Key 与模型配置

本轮沿用现有模型调用临时 Key 机制。当前请求头名称来自既有 RAG 模块，但在本轮中它表示“本次模型调用使用的临时 Key”，不是 RAG 专属能力。

1. 请求头 `X-RAG-API-Key`：用户临时传入的模型 Key。
2. 请求头 `X-RAG-Provider`：模型供应商，例如 `dashscope`、`openrouter`。
3. 环境变量 `OPENAI_API_KEY`、`DASHSCOPE_API_KEY`：服务端默认 Key。
4. 环境变量 `OPENAI_BASE_URL`、`LLM_MODEL`：模型地址和模型名称。

优先级：

1. 前端请求头传入的 Key 优先。
2. 服务端环境变量兜底。
3. 都不存在时不报 500，直接走规则解析。

### FR-07 既有 RAG+LLM 问答能力边界确认

本条不是本轮新增开发目标，而是对系统已有辅导能力的边界确认。现有 RAG 问答能力继续保留：

1. 教师上传课程资料。
2. 系统分块、向量化、存储。
3. 学生提问时向量检索 Top-K 片段。
4. LLM 基于召回片段生成回答。
5. 返回引用来源。

本轮不重写该模块，只在论文和功能说明中强化真实口径：系统辅导侧已经具备基于课程资料召回片段的可溯源问答能力。

## 6. 非功能需求

### NFR-01 稳定性

LLM 能力不能影响现有基础业务。即使 LLM 完全不可用，大纲解析和试题解析仍应尽量通过规则解析工作。

### NFR-02 兼容性

前端现有调用入口尽量不变。响应新增字段不得破坏已有页面逻辑。

### NFR-03 安全性

1. 不在日志中打印完整 API Key。
2. 不把用户通过请求头传入的 Key 持久化到数据库。
3. 不信任 LLM 输出，所有字段必须后端校验。
4. 不允许 LLM 直接决定数据库写入，试题仍需教师确认。

### NFR-04 可观测性

后端应记录以下信息：

1. LLM 解析成功或失败。
2. 失败原因摘要。
3. 是否发生规则回退。
4. 解析出的章节数、课时数或题目数。

日志不得包含敏感 Key 和过长原文。

本轮只记录服务端日志，不新增持久化 LLM 调用记录表。

### NFR-05 性能

1. 单次上传文件大小维持 2MB。
2. LLM 调用应设置明确 HTTP 超时，建议 10 秒到 15 秒，避免请求长期挂起。
3. 若 LLM 超时，应尽快回退规则解析。

## 7. 验收标准

### AC-01 大纲解析成功路径

给定一份结构清晰的 `.md` 大纲，配置有效 LLM Key 后：

1. 接口返回 200。
2. `parseMode` 为 `llm`。
3. 返回章节和课时数组。
4. 所有 `type` 均为 `VIDEO` 或 `TEXT`。

说明：成功路径允许通过真实模型服务、mock LLM 服务或固定样例响应验证，避免外部网络波动影响验收。

### AC-02 大纲解析回退路径

不给 LLM Key 或故意配置错误 Key：

1. 接口不能因为 LLM 失败直接 500。
2. 规则解析成功时返回 200。
3. `parseMode` 为 `rule_fallback`。

### AC-03 试题解析成功路径

给定一个 `.txt` 或 `.md` 文件，文件内容为自然语言要求，例如“生成 3 道 Go 并发单选题，每题 3 分”：

1. 接口返回 200。
2. `parseMode` 为 `llm`。
3. 返回 3 道结构化题目。
4. 每道题包含合法 `type`、`stem`、`options`、`answer`、`score`。

说明：成功路径允许通过真实模型服务、mock LLM 服务或固定样例响应验证，避免外部网络波动影响验收。

### AC-04 试题解析回退路径

不给 LLM Key 或 LLM 返回非法 JSON：

1. 接口回退规则解析。
2. 若输入本身是规则可识别题目文本，返回结构化题目。
3. `parseMode` 为 `rule_fallback`。

### AC-05 教师确认后入库

LLM 解析出的题目不得自动入库。必须经过现有确认导入接口后，才写入 `exam_questions`。

## 8. 测试范围

本轮至少需要覆盖：

1. 后端编译测试：`go test ./...` 或至少 `go test` 覆盖受影响包。
2. 大纲 LLM 成功测试。
3. 大纲 LLM 失败回退测试。
4. 试题 LLM 成功测试。
5. 试题 LLM 失败回退测试。
6. 前端构建或类型检查，确保新增请求头不破坏页面。

如果因为网络或 Key 不可用无法实测 LLM，应至少通过 mock、错误 Key 或无 Key 验证回退路径。

## 9. 论文口径约束

论文中建议使用以下真实表述：

1. “系统实现了基于 LLM 的课程大纲结构化解析，并在 LLM 不可用时回退至规则解析，提升导入功能的稳定性。”
2. “系统实现了 LLM 辅助试题结构化生成/解析，教师可在前端确认和修订后批量导入题库。”
3. “系统通过 RAG 检索课程资料，并调用 LLM 基于召回片段生成可溯源问答结果。”
4. “评估侧提供考试评分、作业批改和结果统计；辅导侧提供课程知识库问答与教师辅助出题，形成初步评-辅闭环。”

论文中不建议继续使用以下表述，除非后续真的实现：

1. “自动识别薄弱知识点并生成专项练习。”
2. “全自动个性化学习路径推荐。”
3. “错题知识点归因与自适应练习推送。”
4. “JSON Schema 强制校验”，除非代码中真的引入 JSON Schema 校验库。
5. “SELECT ... FOR UPDATE 原子化评分”，除非考试提交逻辑真的改为事务加锁。

## 10. 风险与控制

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| LLM 返回格式不稳定 | 解析失败或脏数据进入前端 | 只接受 JSON；后端字段校验；失败回退规则解析 |
| LLM 服务不可用 | 教师无法使用 AI 导入 | 无 Key、超时、错误响应均回退规则解析 |
| 生成题目质量不稳定 | 影响考试质量 | 教师确认后入库；保留置信度和问题提示 |
| 临时功能扩张 | 增加 bug 和论文风险 | 本轮不做自动薄弱点练习生成 |
| 前端无法传递临时模型 Key | LLM 功能不可用 | 沿用 `X-RAG-API-Key`、`X-RAG-Provider` 请求头，并在文档中说明其为历史命名 |
