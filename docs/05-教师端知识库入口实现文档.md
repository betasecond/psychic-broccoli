# 05 教师端知识库入口实现文档

## 1. 文档目标

本文档用于说明本项目教师端“上传/管理知识库文档”入口的实现方式、页面位置、交互流程与验收点。

该入口不是独立后台系统，而是嵌入在教师课程详情页中，确保教师围绕“某一门课程”直接完成知识库管理。

## 2. 入口设计原则

教师端知识库入口遵循以下原则：

1. 入口必须绑定到具体课程。
2. 上传动作必须明确落到当前课程。
3. 管理动作应与课程内容管理放在同一页面。
4. 交互应足够轻量，便于答辩展示。

因此，本项目不新增复杂的“全局知识库后台”，而是采用“课程详情页内嵌知识库管理卡片”的方案。

## 3. 页面入口位置

教师端页面入口位于：

- `frontend/src/pages/teacher/courses/TeacherCourseViewPage.tsx`

教师访问路径为：

1. 进入教师课程列表页。
2. 点击某门课程的“查看”。
3. 进入课程详情页。
4. 在页面中部看到“知识库管理”卡片。

该入口与“课程基本信息”“章节结构”“课程统计”处于同一上下文，符合教师真实使用场景。

## 4. 页面功能组成

“知识库管理”卡片包含以下内容：

1. 上传按钮。
2. 文档格式说明。
3. 已上传知识库文档表格。
4. 删除文档操作。

### 4.1 上传能力

上传按钮调用：

- `ragService.uploadDocument(courseId, file)`

上传后后端完成：

1. 文档解析。
2. 文本分块。
3. 向量化。
4. 写入 `rag_documents` 与 `rag_chunks`。

### 4.2 列表能力

列表调用：

- `ragService.listDocuments(courseId)`

表格展示字段建议至少包括：

1. 文件名。
2. Chunk 数量。
3. 上传时间。

### 4.3 删除能力

删除调用：

- `ragService.deleteDocument(courseId, docId)`

删除动作会同步删除对应文档及其知识块记录。

## 5. 前端服务层设计

服务层文件位于：

- `frontend/src/services/ragService.ts`

教师端依赖的方法包括：

1. `uploadDocument(courseId, file)`
2. `listDocuments(courseId)`
3. `deleteDocument(courseId, docId)`

这样学生端与教师端共用同一套 RAG 服务层，避免接口定义分裂。

## 6. 与学生端的联动关系

教师端知识库入口的作用不是独立存在，而是直接支撑学生端问答。

完整联动链路如下：

1. 教师上传课程文档。
2. 系统构建课程知识库。
3. 学生进入课程详情页提问。
4. 系统从该课程知识库检索片段并生成回答。

因此，教师端入口本质上是学生端 RAG 问答的“建库前置入口”。

## 7. 为什么选择课程详情页作为入口

将教师端知识库入口放在课程详情页，有以下优点：

1. 课程上下文明确，不会误传到错误课程。
2. 教师更容易理解“这批资料属于哪门课”。
3. 与课程章节、课程统计并列展示，演示时逻辑完整。
4. 不需要额外维护一套全局后台导航与状态。

这比单独新增一个“RAG 管理后台”更符合当前项目的规模和答辩目标。

## 8. 当前实现边界

教师端入口当前支持：

1. 单文件上传。
2. 查看文档列表。
3. 删除文档。

教师端入口当前不支持：

1. 批量上传。
2. PDF 解析预处理。
3. 文档内容预览。
4. 分块结果预览。
5. 上传进度百分比显示。
6. 问答质量统计分析。

这些内容可作为后续增强项，但不属于本次教师端入口验收范围。

## 9. 验收标准

教师端入口达到可验收状态时，应满足以下条件：

1. 教师可以在课程详情页直接看到“知识库管理”入口。
2. 教师可以上传 `.txt`、`.md`、`.docx` 文档。
3. 上传成功后，文档会出现在列表中。
4. 文档列表能展示基本元信息。
5. 教师可以删除某个知识库文档。
6. 删除后学生端不再能检索到该文档对应内容。
7. 上传失败、权限不足、格式不支持时有明确提示。

## 10. 建议演示流程

教师端入口的答辩演示可按以下顺序进行：

1. 教师打开课程列表。
2. 进入某门课程详情页。
3. 展示“知识库管理”卡片。
4. 上传一份课程资料。
5. 刷新或自动加载后展示文档列表。
6. 切换到学生端课程详情页。
7. 发起课程问答并展示来源。

这样可以把教师建库与学生问答串联成一条完整业务链路。

## 11. 结论

教师端上传/管理知识库文档入口已经具备明确的页面位置、清晰的课程上下文以及稳定的最小交互链路。

它的核心价值不在于后台复杂度，而在于：

“让教师能够为具体课程建立知识库，并立即服务于学生端 RAG 问答”

这也是本项目 RAG 模块能够成立的关键前提。
